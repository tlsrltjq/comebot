# 2026-06-05 청산 구조 스윕 — 신호 문제 확정 (어떤 TP/SL도 엣지 못 만듦)

## 배경

[2026-06-05 Java parity 백테스트](2026-06-05-java-parity-backtest-no-gross-edge.md)에서
현행 TP4/SL2 설정이 gross PF<1.0(비용 전부터 적자)임을 확인했다. "특정 청산 설정의 문제인가,
신호 자체의 문제인가"를 가르기 위해 parity 하니스에 **TP×SL 스윕**을 추가해 전 구간을 탐색했다.

- 도구: `ParameterSweepTest` (`./gradlew test -Dbacktest.sweep=true --tests "*ParameterSweepTest"`)
- 엔진: 운영 Java 리플레이, **시장가 진입**(maker 역선택 배제 → 신호+청산 순수 엣지 측정)
- 진입(고정): 현행 .env Upbit (1m×20, minPriceChg 0.15, dist 0–5%, minAmt 1M)
- 비용: maker+taker+slip 각 0.05%. 분리: train 120d / test 60d
- 게이트: test PFgross ≥ 1.15 且 |train−test| < 0.15

## 결과 — TP × SL 그리드 (PFgross train/test)

| TP\SL | −1.5 | −2.0 | −3.0 |
|---|---|---|---|
| **+2.0** | 0.921 / 0.900 | 0.896 / 0.923 | 0.888 / 1.023 |
| **+3.0** | 0.925 / 0.947 | 0.909 / 0.937 | 0.890 / 1.049 |
| **+4.0** | 0.937 / 0.957 | 0.895 / 0.961 | 0.880 / 1.030 |
| **+6.0** | 0.913 / 0.944 | 0.898 / 0.966 | 0.896 / 0.929 |

(net PF는 전부 더 낮음. 예: TP4/SL2 = net 0.801/0.860)

## 해석

- **train PFgross가 12개 전부 0.88~0.94 — 1.0 미달.** 신호+청산의 in-sample gross 엣지가 없다.
- test에서 1.0을 넘는 3칸(SL−3.0 열)은 train 0.88~0.89로 **test≫train = 60일 구간 과적합**.
  넓은 SL이 특정 test 윈도우의 되돌림만 살리고 train에선 무효 → 전형적 과적합 시그니처. 게이트 통과 0개.
- 승률은 TP/SL 기하로 깔끔히 트레이드오프(TP2/SL3 60.5% ↔ TP6/SL1.5 19.1%)하나 PF는 개선 없음.

## 결론 / 다음

- **신호 문제 확정.** 청산 구조 튜닝(TP/SL)으로는 현행 풀백 진입을 흑자로 못 만든다. TP4/SL2 고유 문제 아님.
- 레버를 **진입 신호**로 이동: 새 피처/필터 또는 다른 전략 가설을 이 하니스의 PFgross(1차 게이트)로 선별,
  통과 후보만 PFnet의 train/test OOS로 채택 판정. 진입 가설이 train gross ≥ ~1.15을 못 내면 폐기.
- 실거래 전환 금지선(ADR-001) 유효 — 현재 어떤 설정도 수익 모델 아님.
