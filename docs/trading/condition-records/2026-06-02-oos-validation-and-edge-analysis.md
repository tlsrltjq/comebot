# 2026-06-02 OOS 검증 — 시간 필터 과최적화 확인 및 엣지 부재 분석

## 배경

직전(같은 날) 시간대 진입 필터(`0,12,17,21,23`)와 IRYS 제외를 적용했으나,
그 근거는 **전체 180일 데이터로 도출**한 in-sample 최적화였다. 과최적화 위험을 직접
검증하기 위해 train/test 분리(walk-forward)로 재검증했다.

- 데이터: 180일(2025-11-23 ~ 2026-05-22), 1분봉, KRW 상위 10마켓
- 분리: train = 앞 ~120일, test = 뒤 ~60일 (split 2026-03-23)
- 파라미터: 현 운영 `.env` (TP 2.0 / SL -1.5 / 트레일 2.0·1.5)
- 비용: 수수료 0.05%/leg + 슬리피지 0.05%/leg

## 결과 1 — 전략은 두 기간 모두 손실

| 구간 | 거래 | 승률 | PF(slip) | pnl_slip |
|---|---:|---:|---:|---:|
| train | 1,900 | 45.8% | 0.896 | -181,698 |
| test | 977 | 42.6% | 0.785 | -204,891 |

전 마켓 PF < 1.0 (full): ETH 0.963, USDT 0.907, BTC 0.925, DOGE 0.896, PROVE 0.854, TRAC **0.572**, IRYS **0.657**.

## 결과 2 — 시간 필터는 OOS에서 무효 (과최적화 확정)

| 적용 | test 거래 | test 승률 | test PF | test pnl_slip |
|---|---:|---:|---:|---:|
| 필터 없음 | 977 | 42.6% | 0.785 | -204,891 |
| **train 도출 시간** `[0,9,12,14,16,17,18,21,22,23]` → test 적용 | 452 | 41.4% | **0.747** | -113,877 |
| 하드코딩 `[0,12,17,21,23]` → test 적용 | 199 | 51.3% | 1.113 | +18,642 |

- **정직한 OOS 검증(train→test)은 PF 0.747로, 필터 없을 때(0.785)보다 나쁘다.** 시간대 효과는 노이즈.
- 하드코딩 `[0,12,17,21,23]`가 test에서 좋아 보이는 건 **전체기간(=test 포함)으로 골랐기 때문**(in-sample 누수). OOS 아님.

→ **조치: `strategy.entry.allowed-hours-kst` 기본값을 빈 값(비활성)으로 되돌림.** 메커니즘·테스트는 튜닝 도구로 유지.

## 결과 3 — 진입조건 강화로도 OOS 흑자 불가 (스윕)

TRAC/IRYS 제외, 시간필터 없음 기준, 진입조건 그리드 스윕 (test PF, 슬리피지 후):

| config | train PF | test PF | test pnl_slip |
|---|---:|---:|---:|
| 현재(min_vol=0,min_px=0.15) | 0.896 | 0.815 | -151,619 |
| min_vol=100 | 0.892 | 0.840 | -123,640 |
| min_vol=200 | 0.877 | 0.846 | -115,943 |
| min_px=1.0 | 0.897 | 0.779 | -127,891 |
| min_px=2.0 | 0.922 | **0.581** | -100,943 |
| min_vol=100,min_px=1.0 | 0.897 | 0.807 | -103,408 |
| min_vol=50,min_px=1.0,dist[0.5,2] | 0.930 | 0.773 | -67,193 |

- **어떤 조합도 test PF < 1.0** (최고 0.846).
- `min_px`(펌프 강도) 강화는 train만 좋아지고 test는 급락 → 과최적화.
- `min_vol`(거래량 급증) 요구는 train/test 모두 소폭 개선 + 거래수 감소(수수료 절감). 유일하게 견고한 방향이나 흑자 전환엔 부족.

## 근본 원인 — gross edge < 비용

- 수수료 전(raw) 순이익 +189,000 KRW / 2,877거래 = **거래당 +0.066%**
- 왕복 수수료만 0.1% (Upbit 0.05%×2), 슬리피지 포함 시 0.2%
- **gross edge 0.066% < 비용 0.1%** → 거래할수록 손실. 파라미터 튜닝으로 해결 불가.

## 적용한 변경

| 항목 | 변경 |
|---|---|
| `strategy.entry.allowed-hours-kst` 기본값 | `0,12,17,21,23` → **빈 값(비활성)** |

유지: TRAC/IRYS 제외(OOS에서도 견고하게 최악 마켓), 시간 필터 메커니즘·테스트(튜닝 도구).
미변경: DOGE/PROVE 제외 — OOS에선 특별히 우량하지 않아(직전 기록의 "PROVE 최고"는 in-sample 누수였음) 현 목록 유지.

## 권장 .env 설정 (운영자 판단)

- `STRATEGY_CANDIDATE_UPBIT_MIN_TRADE_AMOUNT_CHANGE_RATE=100` (현재 0) — 거래량 급증 요구로 과매매·수수료 드래그 감소. 흑자 전환은 아니나 손실 축소 + 건전성.

## 결론 및 다음 방향

현 pullback-bounce + TP/SL/trailing 구조는 **비용 차감 후 검증된 엣지가 없다.** 파라미터 튜닝은 답이 아니다. 실거래 금지(PAPER 유지). 다음은 §PROJECT 우선순위 참조:

1. **비용 구조 개선** — 거래당 gross edge(0.066%)가 왕복 비용(0.1%)을 넘어야 함. 거래 빈도 대폭 축소 또는 maker 주문.
2. **청산 구조 재설계** — 현 trailing은 승자를 +1.1%로 캡. 승자 run 허용(넓은 TP/시간기반)으로 평균 승 폭 확대.
3. **신호 timeframe 재검토** — 1분봉 pullback은 노이즈 비중이 큼. 5분/15분 등 상위 timeframe 실험.
4. **데이터 누적 후 재검증** — 모든 변경은 train/test OOS로만 채택.
